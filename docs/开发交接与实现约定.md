# 开发交接与实现约定

本文档是后续开发默认约定，目的是减少重复确认与无意义沟通。

## 1. 已确认技术决策（固定）

- 后端：FastAPI + Celery
- Embedding：BGE（默认）
- LLM：小米 MiMo（默认）
- Web：Next.js + Ant Design starter（方案3）
- 中间件：PostgreSQL + Redis + ChromaDB（统一 Docker Compose 管理）

## 2. 配置与边界

- Provider 名称、模型名、端点、密钥均来自配置文件和环境变量。
- 业务逻辑禁止硬编码 provider 细节。
- PostgreSQL 是元数据事实源；向量库数据可重建。
- API 错误响应必须包含 `error.code` 和 `request_id`（后续实现时严格遵守）。

## 3. 当前骨架范围

已完成：

- 后端分层骨架目录与健康检查接口
- Celery app 配置
- 数据库模型（7 表）+ Alembic 迁移
- 知识库 CRUD + 文档上传 API（含鉴权、去重、对象存储抽象）
- 异步索引流水线（解析器/切分器/BGE/Chroma/幂等重试）
- RAG 查询链路（检索/Prompt/MiMo LLM/持久化/来源引用）
- 统一错误响应（error.code + request_id）
- 请求 ID 中间件和全局异常映射
- Web 项目基础目录和运行脚本
- Docker Compose 本地编排

未完成（留给后续开发）：

- Web UI 三大核心功能页（KB、文档、问答）
- MiMo/BGE 实际 API 调试验证（代码已就绪，待配置真实端点）
- PDF/DOCX 解析器
- 流式查询接口（SSE）
- 测试与可观测性补齐

## 4. 后续开发顺序（建议）

1. 数据库模型 + Alembic 迁移
2. 知识库与文档上传 API
3. 索引 Worker（解析/分块/BGE/Chroma）
4. 查询 API（检索 + MiMo + 引用）
5. Web UI 三条主流程对接
6. 测试与可观测性补齐

## 5. 目录约定（执行标准）

- 后端：`app/api`、`app/application`、`app/domain`、`app/infrastructure`、`app/tasks`
- 前端：`web/src/app`、`web/src/features`、`web/src/services`、`web/src/hooks`、`web/src/types`
- 文档：`docs/`；术语统一文件：`CONTEXT.md`

## 6. 协作规则

- 新增术语、状态、错误码时，同步更新 `CONTEXT.md`。
- 任何跨层改动，先更新接口契约再改实现。
- 需求发生变化时，以“先改文档后改代码”为默认流程。
