# 开发实操指南（你来实现）

目标：你按本文档一步步做，即可在不反复沟通的情况下，把当前骨架推进到可用 MVP。

---

## 0. 先读这些文件（5 分钟）

开始前只看这 4 个文件，避免跑偏：

1. `docs/开发交接与实现约定.md`
2. `docs/RAG系统项目架构设计文档.md`
3. `CONTEXT.md`
4. `.trellis/spec/backend/*.md` 与 `.trellis/spec/frontend/*.md`

结论必须保持一致：

- Embedding 默认 BGE
- LLM 默认 MiMo（配置驱动）
- Web 使用 Next.js + Ant Design（方案 3）
- 中间件统一由 Docker Compose 管理

---

## 1. 每次开始开发的固定流程

### 1.1 启动环境

```bash
cp .env.example .env
docker compose up --build
```

检查：

- `GET http://localhost:8000/api/v1/health` 返回 `{"status":"ok"}`
- 打开 `http://localhost:3000` 可看到 Web scaffold 页面

### 1.2 开发顺序纪律

每个功能都按这 4 步：

1. 先定义接口契约（请求/响应/错误码）
2. 再写后端实现
3. 再接前端页面
4. 最后补测试与文档

禁止顺序：先堆页面再反推 API。

---

## 2. 任务拆解（按优先级）

## P0-1：数据库与迁移

你需要完成：

- SQLAlchemy 模型（users、knowledge_bases、documents、chunks、indexing_jobs、query_logs、query_sources）
- Alembic 初始化与首个迁移
- 会话管理与仓储基础

完成标准：

- `alembic upgrade head` 成功
- PostgreSQL 中可看到以上表结构
- 本地重建库流程稳定可复现

---

## P0-2：知识库与文档上传 API

你需要完成：

- `POST /api/v1/knowledge-bases`
- `GET /api/v1/knowledge-bases`
- `POST /api/v1/documents/upload`
- `GET /api/v1/documents/{id}`
- checksum 去重逻辑

完成标准：

- 能创建 KB
- 能上传 TXT/MD 并生成 Document 记录
- 重复上传同文件可识别
- 错误响应包含 `error.code` 与 `request_id`

---

## P0-3：索引链路（异步）

你需要完成：

- Celery indexing task
- 文档解析（先 TXT/MD）
- 文本切分器
- BGE provider 实际接入
- Chroma 向量入库
- 文档状态流转：`uploaded -> processing -> indexed/failed`

完成标准：

- 上传后异步进入 `indexed`
- 失败有明确错误码和日志
- 重试不会重复写脏数据（幂等）

---

## P0-4：查询链路（RAG）

你需要完成：

- RetrievalService
- PromptBuilder
- MiMo provider 实际接入（读取配置）
- `POST /api/v1/queries`
- 返回 `sources`

完成标准：

- 对已索引 KB 提问可得到答案
- 答案包含来源片段
- 无结果时明确拒答或说明无法确定

---

## P1：Web UI 三条主流程

必须先做这三页，其他都后置：

1. 知识库列表/创建页
2. 文档上传与索引状态页
3. 问答与引用展示页

完成标准：

- 从 UI 可走通：创建 KB -> 上传 -> 等待 indexed -> 提问 -> 展示答案+引用

## 3. 文件落点规范（避免你后面重构）

后端：

- 路由：`app/api/v1/routes/*.py`
- 用例编排：`app/application/services/*.py`
- 领域规则与错误：`app/domain/*.py`
- 基础设施适配：`app/infrastructure/**/*`
- 任务：`app/tasks/*`

前端：

- 页面：`web/src/app/*`
- 业务模块：`web/src/features/*`
- API 调用：`web/src/services/*`
- 状态管理：`web/src/stores/*`
- 类型：`web/src/types/*`

---

## 4. 每完成一个功能必须做的检查清单

- [ ] API 是否有明确输入/输出类型
- [ ] 错误码是否稳定（不是临时字符串）
- [ ] 日志是否带 `request_id` 或 `job_id`
- [ ] 配置是否全部来自 `.env` / config
- [ ] 是否写了最小测试（至少 happy path + 1 个失败场景）
- [ ] 文档是否同步更新（接口、状态、术语）

---

## 5. 文档更新规则（非常重要）

当你做了以下变更时，必须同步文档：

- 新增术语/状态：更新 `CONTEXT.md`
- 新增错误码/响应结构：更新 `docs/RAG系统项目架构设计文档.md`
- 变更目录或分层边界：更新 `.trellis/spec/*/directory-structure.md`
- 变更协作约定：更新 `docs/开发交接与实现约定.md`

一句话：**代码变，文档必须同一提交更新。**

---

## 6. 建议你的提交节奏

推荐每个子任务单独提交：

1. `feat(db): add initial models and alembic migration`
2. `feat(api): add knowledge base and document upload endpoints`
3. `feat(worker): implement indexing pipeline with bge`
4. `feat(query): add retrieval and mimo generation`
5. `feat(web): implement kb upload query core pages`
6. `docs: update context and implementation notes`

这样你回滚和排错都会轻松很多。

---

## 7. 卡住时如何自救（先做这三步）

1. 看 `.trellis/spec/` 对应层规范，确认是否偏离约定
2. 用最小输入复现（例如最小 TXT 文件）
3. 先打通主链路，再补优化（不要同时修三层）

---

## 8. 最终验收口径（你做到这些就算 MVP 可演示）

- Docker Compose 一键启动
- 可创建 KB
- 可上传并索引文档
- 可发起查询并返回引用来源
- Web UI 能完整走通主流程
- 关键日志和错误码可排查

达到以上即进入下一阶段（检索优化/评测体系），否则不扩功能。
